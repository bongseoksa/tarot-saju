## 프론트엔드: React + Vite

### 선정 이유
- **Capacitor 호환성**: SPA 빌드 결과물을 그대로 WebView에 로드 가능. Next.js는 `output: 'export'` 설정이 필요하며 이 경우 SSR, 미들웨어, API Routes 등 핵심 기능을 사용할 수 없음
- **빌드 속도**: Vite의 ESBuild 기반 빌드가 Next.js 대비 빠름. MVP 개발 사이클에 유리
- **단순한 구조**: 서버/클라이언트 컴포넌트 경계 구분 불필요. 모든 코드가 클라이언트에서 실행됨
- **번들 사이즈**: Next.js 프레임워크 오버헤드 없이 가벼운 결과물

### Next.js를 선택하지 않은 이유
| Next.js 장점 | 이 프로젝트에서의 필요성 |
|---|---|
| SSR/SSG | Capacitor WebView 안에서 실행 → SSR 무의미. SEO 필요 페이지는 프리렌더로 대응 |
| SEO | SPA + 프리렌더 + Edge 동적 OG로 충분. 아래 SEO 전략 참조 |
| API Routes | Supabase Edge Functions로 대체 |
| 서버 컴포넌트 | 카드 뽑기, AI 응답 등 클라이언트 인터랙션 중심 |

> 웹 서비스 병행이 결정되었으나, SEO가 필요한 페이지가 제한적이므로 Next.js 전환 비용 대비 효과가 낮다고 판단. 검색 유입이 유의미해지는 시점에 재검토.

### SEO 대응 전략 (SPA + 프리렌더)

웹 서비스를 병행하므로 SEO가 필요하나, SSR 프레임워크(Next.js) 없이 대응 가능.

#### SEO가 필요한 페이지 vs 불필요한 페이지

| 페이지 | SEO 필요 | 이유 |
|---|---|---|
| 홈 (테마 목록) | O | "타로 운세", "오늘의 타로" 검색 유입 |
| 테마별 랜딩 | O | "연애 타로", "이직 타로" 롱테일 키워드 |
| 공유 결과 페이지 | O | 카카오/인스타 공유 시 OG 메타태그 필수 |
| 카드 뽑기 | X | 인터랙션 중심, 크롤러 불필요 |
| AI 해석 결과 | X | 동적 생성, 개인화 콘텐츠 |

#### MVP 대응 방식

1. **정적 프리렌더** — vite-ssg로 홈(`/`) 1페이지만 빌드 타임에 HTML 생성
   - title: "점하나 - 오늘, 점 하나 찍어볼까?"
   - description: "AI가 읽어주는 타로 카드. 무료, 무제한. 지금 바로 점 하나 찍어보세요."
   - OG 이미지: 점하나 대표 이미지 (정적)
2. **동적 OG 메타태그** — `/shared/:shareId` 공유 결과 페이지는 Supabase Edge Function에서 OG 태그 동적 주입
3. **기본 SEO** — sitemap.xml (vite-ssg 자동 생성), robots.txt (크롤링 허용)

## 스타일링: Tailwind CSS

### 선정 이유
- **빠른 UI 개발**: 클래스 기반으로 별도 CSS 파일 관리 불필요
- **일관된 디자인 시스템**: 기본 제공되는 spacing, color, typography 스케일
- **번들 최적화**: 사용하지 않는 스타일 자동 제거 (PurgeCSS)
- **v0/AI 도구 호환**: AI 기반 UI 생성 도구들이 Tailwind 코드를 출력

## 상태 관리

### Zustand 스토어

| 스토어 | 역할 | 영속성 |
|---|---|---|
| `useReadingStore` | 현재 뽑기 세션 (선택된 카드, 테마ID, 진행 상태) | 메모리만 (페이지 이탈 시 초기화) |
| `useHistoryStore` | 완료된 결과 목록 (최대 20건) | localStorage (암호화+압축) |
| `usePendingStore` | 미완료 세션 (최대 3건, 당일 자정 만료) | localStorage (암호화+압축) |

- 스토어 분리 기준: 영속성 (메모리 vs localStorage)
- 카테고리 필터: URL 쿼리 파라미터로 관리, 스토어 불필요

### localStorage 관리

- **스토리지 유틸 단일 모듈** (`storageUtil`): 암호화(AES) + 압축(lz-string) + 읽기/쓰기를 하나로 통합
- Zustand `persist` 미들웨어: 커스텀 storage로 스토리지 유틸 사용
- 모달 억제 플래그 (`suppressedUntil`): 스토어 밖에서 동일한 스토리지 유틸로 직접 관리 (미완료 세션과 독립, 당일 자정 만료)
- 모든 localStorage 접근은 스토리지 유틸 경유 — 직접 `localStorage.getItem/setItem` 호출 금지

## localStorage 정책

### 저장 대상
- 결과 히스토리: 최대 20건
- 미완료 세션: 최대 3건 (당일 자정 만료)

### 암호화 + 압축
모든 localStorage 데이터에 암호화 + 압축 적용.
- 처리 순서: `JSON → lz-string 압축 → AES 암호화 → base64 → localStorage`
- 읽기 순서: `localStorage → base64 → AES 복호화 → lz-string 해제 → JSON`
- 목적: 개발자 도구 평문 노출 방지 + 용량 절감

## 반응형 레이아웃

> 레이아웃 리서치 상세: `docs/specs/deep-research-report.md`

### 레이아웃 조합 원칙

```
허브형 홈 → 집중형 카드 뽑기 → 광고 게이트 → 리포트형 결과 → 공유/히스토리
```

- 홈: 허브형 — 퀵 진입 CTA + 카테고리 칩 + 테마 카드 목록
- 카드 뽑기: 집중형 — 최소 장식, 선택에 집중
- 결과: 리포트형 — 카드 요약 → 한줄 결론 → 섹션별 본문 (접기/펼치기)
- 채팅형은 결과 첫 2~3문단 순차 노출에만 제한 적용

### 반응형 브레이크포인트 (KRDS 표준형)

| 브레이크포인트 | 컬럼 | 가터 | 마진 | 적용 |
|---|---:|---:|---:|---|
| 360px+ (sm) | 4 | 16px | 16px | 홈, 카드 선택, 결과를 1열 중심 설계 |
| 768px+ (md) | 8 | 16px | 24px | 리스트와 본문을 2영역으로 분할 |
| 1024px+ (lg) | 12 | 24px | 24px | 결과 본문 + 요약 패널, 필터 + 콘텐츠 |
| 1280px+ (xl) | 12 | 24px | 24px | 중앙 max-width, 좌우 보조 패널 |

- 모바일을 데스크톱으로 단순 확대하지 않음
- 데스크톱 여백은 요약/필터/히스토리 패널에 활용
- 폼/입력은 단일 열 수직 정렬 우선

### 화면별 레이아웃

#### 홈

| 구분 | 모바일 | 데스크톱 (lg+) |

| 헤더 | 로고 + 히스토리 아이콘 | 좌: 로고, 중앙: 카테고리, 우: 히스토리 |
| 첫 화면 | 퀵 CTA 2개 + 카테고리 칩 + 테마 카드 리스트 | 좌: 카테고리 필터, 중앙: 테마 카드 그리드, 우: 최근 결과/배너 |

| 구분 | 모바일 | 데스크톱 (lg+) |
|---|---|---|
| 상단 | 카드 3장 요약 + 한줄 결론 + 공유 버튼 | 좌: 목차 패널, 중앙: 본문, 우: 카드 요약/공유/저장 sticky |

| 하단 | 공유 + "점 하나 더 찍어볼까?" CTA | 동일 |

## 디자인 가이드

### 톤 & 무드

**무드 키워드:** 가벼운 신비감 / 구조적 가독성 / 한국형 에디토리얼 UI

| 요소 | 방향 |
|---|---|
| 전체 톤 | 밝은 베이스 — 친근하고 부담 없는 진입, 밤의 감성은 은은하게 |
| 무게감 | 구조화된 카드/리스트/리포트 위계가 먼저, 장식은 부차적 |
| 홈 화면 | 첫 행동(퀵 CTA)이 강하게 보이고, 나머지는 아래로 |
| 결과 화면 | 에디토리얼 리딩 인터페이스 — 읽기 좋은 타이포, 차분한 본문 |
| 카드 선택 화면 | 집중형 — 최소 장식, 카드 선택에 몰입 |

### 레퍼런스 방향
- 토스/당근 같은 밝고 친근한 앱 톤 + 카테고리별 감성 포인트
- 별자리 앱(Co-Star, The Pattern)의 미니멀 감성 참고

### 피해야 할 것
- 금색 테두리, 한자, 부적 이미지 등 전통 점술 클리셰
- 과도한 이펙트 (반짝이, 별 파티클, 네온 글로우 등)
- 전체 다크 모드 (가벼운 진입 톤과 불일치)
- AI스러운 빛나는 패널, 게임 UI 느낌
- 모바일을 그대로 늘린 데스크톱 레이아웃
- 광고를 콘텐츠 카드처럼 꾸미는 것

### 카드 선택 화면 -- 카테고리별 톤

카드 선택 진입 시 배경이 카테고리 색감으로 부드럽게 전환. 밝은 톤 안에서의 변화라 위화감 없음.

| 카테고리 | 분위기 | 색감 방향 |
|---|---|---|
| 일상 | 따뜻한, 편안한 | 소프트 옐로우 / 베이지 |
| 연애 | 설레는, 부드러운 | 소프트 핑크 / 로즈 |
| 직장 | 차분한, 신뢰감 | 소프트 블루 |
| 재물 | 풍요로운 | 소프트 그린 / 민트 |
| 학업 | 집중, 맑은 | 라이트 스카이블루 |
| 기타 | 신비로운 | 소프트 퍼플 / 라벤더 |

#### 카테고리 배경 (카드 선택 화면)

| 카테고리 | 배경 톤 | 포인트 톤 |
|---|---|---|
| 일상 | 소프트 옐로우 / 베이지 (예: `#FEF9C3`) | 웜 옐로우 (예: `#EAB308`) |
| 연애 | 소프트 핑크 (예: `#FCE7F3`) | 로즈 핑크 (예: `#EC4899`) |
| 직장 | 소프트 블루 (예: `#DBEAFE`) | 미디엄 블루 (예: `#3B82F6`) |
| 재물 | 소프트 그린 / 민트 (예: `#D1FAE5`) | 에메랄드 (예: `#10B981`) |
| 학업 | 라이트 스카이블루 (예: `#E0F2FE`) | 스카이블루 (예: `#0EA5E9`) |
| 기타 | 소프트 퍼플 / 라벤더 (예: `#EDE9FE`) | 바이올렛 (예: `#8B5CF6`) |

> 전체적으로 파스텔 톤 범위 안에서 조율. 카테고리 간 색감 차이가 명확하되, 밝은 톤의 통일감 유지.

### 카드 선택 경험

> 카드 뽑기는 서비스의 핵심 경험. "내가 고른 카드"라는 감각을 만드는 것이 목표.
> 세부 타이밍/이징은 스케치 및 구현 시 보정.

**1단계 -- 고르기 전 (기대감)**
- 카드에 손이 가까워지면(호버/터치) 카드가 살짝 떠오름
- 미세한 빛남 효과 — "이 카드가 나를 부르는 것 같은" 느낌

**2단계 -- 선택 순간 (몰입)**
- 터치 시 카드가 위로 떠오르며 뒤집힘
- 단순 회전이 아닌, 공중에서 천천히 열리는 느낌
- 뒤집히는 동안 주변 카드가 살짝 어두워짐 (선택된 카드에 집중)
- 시각 피드백이 메인: 카드 선택 시 짧은 스케일 펄스(1.05x → 1.0x) + 주변 어두워짐
- 햅틱 피드백: 모바일 앱(Capacitor) 햅틱, 모바일 웹 Vibration API (미지원 시 무시), 데스크톱은 시각만
- 진동 기본 ON, 토글은 설정 페이지(백로그)에서 제공 예정. MVP는 기기 설정으로 대체

**3단계 -- 공개 (설렘)**
- 카드 앞면이 보이면서 부드러운 글로우
- 카드명이 아래에서 페이드인
- 정방향: 카테고리 포인트 색상 글로우
- 역방향: 카드가 거꾸로(180도) 공개 + 부드러운 보라 글로우 + "역방향" 라벨 페이드인

**4단계 -- 자리잡기 (완성감)**
- 카드가 상단 슬롯으로 부드럽게 이동 (스프링 커브)
- 슬롯에 안착하면서 가벼운 바운스
- "과거" / "현재" / "미래" 라벨이 함께 활성화

**구현 방향:** Framer Motion (스프링/제스처 지원)

### 카드 이미지 에셋

**방식: AI 생성 커스텀 (Midjourney/DALL-E)**
- 메이저 아르카나 22장 + 카드 뒷면 1장 = 총 23장
- "밝고 귀여운" 브랜드 톤에 맞춘 일러스트 스타일
- 22장 일관성 유지를 위해 동일 프롬프트 베이스 + 카드별 변수만 변경
- 이미지 규격: 스케치에서 확정 (권장 비율 2:3, WebP 포맷)
- 카드 뒷면: 점하나 로고/심볼 + 카테고리 무관 공통 디자인

**생성 가이드 키워드 (방향):**
- 스타일: flat illustration / cute / pastel / minimal
- 톤: 밝고 따뜻한, 전통 타로 클리셰 배제
- 각 카드의 전통적 상징을 귀여운 일러스트로 재해석

### API 테스트 -- MSW 기반 네트워크 모킹

외부 API(Supabase 등)를 호출하는 코드는 `vi.mock`으로 클라이언트를 직접 모킹하지 않고, `@packages/msw-handler`를 사용하여 네트워크 레벨에서 요청을 가로챈다.

| 구분 | 설명 |
|---|---|
| 패키지 | `packages/msw-handler` (`@packages/msw-handler`) |
| 핸들러 정의 | `createMock({ route, method, resolver })` |
| 테스트 서버 | `packages/msw-handler/src/server.ts` — `setupServer(gatewayHandler)` (MSW Node) |
| 브라우저 지원 | `MSWProvider` + `useMSWContext` (MSW Service Worker) |
| 동적 설정 | `setConfig(mockName, config)` — 테스트별 응답 변경 |

### 선정 이유
- **실제 네트워크 동작 검증**: fetch/XHR을 네트워크 레벨에서 가로채므로 클라이언트 코드 변경 없이 테스트 가능
- **일관된 모킹 패턴**: `createMock` 패턴으로 모든 API 모킹을 통일
- **브라우저/Node 양쪽 지원**: 테스트(Node)와 개발 환경(브라우저) 동일 핸들러 재사용
