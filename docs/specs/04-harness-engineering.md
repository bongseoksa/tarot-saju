# 하네스 엔지니어링 적용 방안

본 프로젝트에서 AI가 관여하는 두 가지 영역에 하네스 엔지니어링을 적용한다.

1. **개발 시점 AI** — Claude Code가 코드를 작성하고 프로젝트를 관리하는 구조
2. **런타임 AI** — Gemma 모델이 타로 해석 문장을 생성하는 구조

---

## 1. 개발 시점 하네스 — Claude Code 제어 구조

Claude Code가 이 저장소에서 작업할 때 올바른 방향으로 동작하도록 감싸는 구조.

### 현재 적용된 하네스

| 하네스 요소 | 구현 위치 | 역할 |
|---|---|---|
| 시스템 프롬프트 | `CLAUDE.md` | 프로젝트 구조, 설계 원칙, 언어 규칙 전달 |
| 스펙 문서 | `docs/specs/*.md` | 기능 요구사항과 설계 결정의 컨텍스트 제공 |
| 문서 최신화 규칙 | `CLAUDE.md` > 작업 후 문서 최신화 | 코드-문서 불일치 방지를 위한 피드백 루프 |

### 추가 적용 방안

#### 1-1. CLAUDE.md 고도화

현재 `CLAUDE.md`는 프로젝트 개요 수준. 코드가 작성되기 시작하면 다음을 추가한다.

- **빌드/테스트/린트 명령어** — Claude Code가 변경 후 즉시 검증할 수 있도록
- **코드 컨벤션** — 파일 네이밍, 컴포넌트 구조, 상태 관리 패턴 등
- **금지 사항** — 직접 DB 스키마 변경 금지 (마이그레이션 필수), env 파일 커밋 금지 등

#### 1-2. 가드레일 (Pre-commit Hook)

Claude Code가 생성한 코드가 자동으로 검증되는 구조.

```
코드 생성 → lint (ESLint) → 타입 체크 (tsc) → 테스트 → 커밋
```

- **ESLint + Prettier** — 코드 스타일 강제
- **TypeScript strict mode** — 타입 안전성 보장
- **Pre-commit hook (husky + lint-staged)** — 커밋 전 자동 검증

#### 1-3. 피드백 루프

Claude Code 작업 결과를 평가하고 개선하는 순환 구조.

```
작업 요청 → Claude Code 실행 → 테스트/빌드 검증 → 실패시 자동 수정 → docs/ 최신화
```

- 테스트 실패 시 Claude Code가 원인을 분석하고 수정하는 루프
- 작업 완료 후 관련 스펙 문서 변경 여부 확인 (이미 CLAUDE.md에 명시)

## 2. 런타임 하네스 — 타로 해석 AI 제어 구조

Gemma 모델이 타로 해석 문장을 생성할 때 품질과 안전성을 확보하는 구조.
이 프로젝트에서 **프롬프트 품질 = 서비스 품질**이므로 런타임 하네스 설계가 핵심.

### 2-1. 입력 하네스 (프롬프트 조립)

AI에게 전달하는 프롬프트를 정형화된 구조로 조립한다. 모델이 "알아서" 판단하는 영역을 최소화.

```
[시스템 프롬프트]          ← 역할 정의, 출력 형식, 제약 조건
  + [카드 정적 데이터]     ← DB/JSON에서 조회 (meaningUpright, meaningReversed, description)
  + [스프레드 규칙]        ← 위치별 의미 (과거/현재/미래)
  + [사용자 질문]          ← 카테고리 + 자유 텍스트
  + [출력 지시]            ← 응답 길이, 말투, 구조
```

**설계 원칙:**
- 카드 의미 해석을 모델에게 맡기지 않는다 — 정적 데이터로 제공
- 스프레드 위치별 역할을 모델에게 추론시키지 않는다 — 명시적으로 전달
- 모델은 "주어진 재료로 문장을 조합"하는 역할만 수행

#### 시스템 프롬프트 템플릿

> 최신 프롬프트는 [03-ai-design.md](./03-ai-design.md)의 "시스템 지시 (고정)" 참조.
> 심리 설계 원칙(바넘 효과, 양면 진술, 감정 레이블링, 성찰 유도, 긍정 전환)이 프롬프트에 반영되어 있음.

### 2-2. 출력 가드레일

AI 응답이 서비스 품질 기준을 충족하는지 검증하는 구조.

| 검증 항목 | 방법 | 실패 시 처리 |
|---|---|---|
| 응답 형식 | `### 카드 해석` / `### 조언` / `### 한줄 요약` 섹션 존재 여부 | 재요청 (최대 1회) |
| 응답 길이 | 최소 50자 / 최대 1000자 | 범위 초과 시 잘라내기 |
| 심리적 납득감 | 바넘 효과·양면 진술·감정 레이블링 등 심리 기법 반영 여부 | 프롬프트 튜닝 |
| 금칙어 | 부적절한 표현, 극단적 예언, 완전한 부정/절망 표현 | 기본 해석으로 대체 |
| 카드 일관성 | 응답에 다른 카드 이름이 포함되지 않는지 | 재요청 |

### 2-3. 폴백 전략

AI 서빙이 실패하는 경우(PC 꺼짐, 모델 오류 등)에 대한 대비.

```
AI 해석 요청 → [타임아웃 30초] → 실패 시 정적 해석 반환
```

- **정적 해석 데이터** — 카드별 미리 작성된 기본 해석 문장을 DB/JSON에 보관
- AI 실패 시 정적 해석을 반환하되, "AI 해석을 불러오지 못했습니다" 안내 표시

### 2-4. 프롬프트 버전 관리

프롬프트 변경이 서비스 품질에 직접 영향을 미치므로 버전 관리한다.

```
/packages/shared/prompts/
  ├── v1-one-card.ts       # 원카드 프롬프트 템플릿
  ├── v1-three-card.ts     # 쓰리카드 프롬프트 템플릿
  └── index.ts             # 현재 활성 버전 export
```

- 프롬프트 변경 시 새 버전 파일 생성 (v1 → v2)
- A/B 테스트를 위해 버전 전환 가능하도록 설계
- 프롬프트 변경 이력을 git으로 추적

## 3. 하네스 적용 우선순위

MVP 단계에서는 최소한의 하네스만 적용하고, 서비스 성장에 따라 확장한다.

### Phase 1 (MVP)

| 항목 | 상태 |
|---|---|
| CLAUDE.md (개발 시점 컨텍스트) | 적용 완료 |
| 문서 최신화 규칙 | 적용 완료 |
| 시스템 프롬프트 템플릿 | **MVP 필수** |
| 출력 형식 검증 | **MVP 필수** |
| 정적 해석 폴백 | **MVP 필수** |
| 프롬프트 버전 관리 | **MVP 필수** |

### Phase 2 (서비스 성장 후)

| 항목 | 조건 |
|---|---|
| Pre-commit hook (husky) | 코드베이스 규모 커질 때 |
| 금칙어 필터링 | 사용자 피드백 기반 |
| 프롬프트 A/B 테스트 | 유저 100명 이상 |
| 응답 품질 로깅/모니터링 | 유료 전환 시작 후 |

## 4. 이 프로젝트에서 하네스가 중요한 이유

이 프로젝트의 구조적 특성상 하네스 엔지니어링이 특히 중요하다.

1. **AI 역할이 한정적** — 해석 문장 생성 하나뿐이므로, 그 하나를 잘 제어하면 전체 서비스 품질이 결정됨
2. **경량 모델 사용** — Gemma는 GPT-4 대비 능력이 제한적. 모델의 부족함을 하네스(정적 데이터 + 구조화된 프롬프트)로 보완해야 함
3. **로컬 서빙의 불안정성** — PC 기반 서빙이므로 폴백 전략이 필수
4. **1인 개발** — Claude Code가 주요 개발 도구이므로 개발 시점 하네스(CLAUDE.md, 스펙 문서)가 코드 품질을 직접 좌우함

> 모델을 바꾸는 것보다 하네스를 개선하는 것이 비용 대비 효과가 크다.
> 특히 비용 0원 원칙을 지키면서 품질을 올리려면, 하네스가 유일한 레버다.
